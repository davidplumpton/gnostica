[1] **Mind Map Format Overview** - A graph-based documentation format stored as plain text files where each node is a single line containing an ID, title, and inline references [2]. It serves as a superset structure that can represent trees, lists, or any graph topology [4], scaling from small projects (<50 nodes) to complex systems (500+ nodes) [5].

Nodes are line-oriented, allowing line-by-line loading and editing by AI models [3]. The inline reference syntax `[N]` creates bidirectional navigation between concepts, with links embedded naturally within descriptive text rather than as separate metadata [1][4]. This structure is both machine-parseable and human-readable, supporting grep-based lookups for quick node retrieval [3].

[3] **Technical Advantages** - The format enables line-by-line overwriting of nodes without complex parsing [2], making incremental updates efficient for both humans and AI agents [1]. LLMs generate this format naturally because citation syntax `[N]` mirrors academic paper references they've seen extensively during training [1][5].

[6] **Gnostica App Overview** - Gnostica is a Clojure/ClojureScript implementation of the tarot-and-Icehouse territory game documented in `docs/rules.txt` [7]. The browser UI is built with Reagent and re-frame in `src/main/gnostica/app.cljs`, with shared pure data helpers in `.cljc` namespaces [8][9][16][17][18][19]. `src/main/gnostica/app_state.cljc` now initializes the re-frame app-db from the authoritative game state so browser subscriptions do not maintain a second board setup path, derives current-player hand/draw/discard card zones from `:game`, owns the top-level `:move-selection` state machine for staged current-player move choices including Cup wasteland territory creation, and routes completed Cup territory/hand-card confirmations through `gnostica.game-state/apply-cup-move` [9][10][17]. Three.js and OrbitControls are available as pinned, SRI-checked CDN globals gated to r128, and `gnostica.three-board` renders the current board as movable, selectable textured card planes over a dark velvet table surface with faint wasteland outlines, visible Icehouse pyramid pieces with black edge outlines, and one/two/three base pip markers without adding npm [11][12][13][14][15][16][18]. Browser keyboard commands are exposed through a `?` help dialog and tracked as UI-only app state [22].

[7] **Rules Source** - `docs/rules.txt` is the current rules reference. The first implemented rule slice is board setup: nine tarot cards are dealt face up into a 3x3 territory grid, and adjacent territory cards alternate between portrait and landscape orientation with no touching edges [8][9].

[8] **Board Model** - `src/main/gnostica/board.cljc` defines the initial 3x3 board. It shuffles the tarot deck before taking nine table cards, records each cell's row, column, `:face :up`, and assigns checkerboard orientation with top-left portrait so every orthogonal neighbor is opposite [7][10]. `initial-board` also accepts an injectable shuffle function so tests can use deterministic deck order [10].

[9] **Board Rendering** - `src/main/gnostica/app.cljs` renders the board as the first screen and owns re-frame subscriptions/events, while app-db initialization and selection helpers live in browser-free `gnostica.app-state`, which stores the `gnostica.game-state/create-game` result under `:game`, derives board cells, on-board pieces, selected territory, current player, current-player hand, draw pile, discard pile, and legal move source availability from that shared value, and keeps staged move UI state outside `:game` under `:move-selection` [8][16][17]. The main layout places the 3D/CSS board above display-only card zones that show current-player hand card images/titles, draw deck count with a facedown deck affordance, and discard pile count plus an empty state or top discarded card; standalone draw/discard still needs a future pure transition, while Cup hand-card moves update `:game` by discarding the source card through `apply-cup-move` [10][17]. The move panel lets the current player choose a source, then stage source territory, hand card, minion, existing target territory, Cup target wasteland, one-point territory card, draw count, orientation, confirmation, and cancellation; completed Cup territory/hand-card confirmations replace `:game` on success for either existing-territory small-piece placement or wasteland territory creation and preserve the staged selection with structured gameplay errors on rejection, while source types without pure transitions still report explicit unavailable results. Three.js card picking and CSS fallback card buttons both dispatch through the shared board-selection event so inspection and existing-territory move selection stay synchronized; Cup wastelands are selected from the move panel [12][14][17]. The normal board surface is `gnostica.three-board/scene`, a Reagent lifecycle component that mounts and disposes a Three.js renderer, OrbitControls, raycast picking listeners, card selection highlight meshes, dark velvet table plane, wasteland outline lines, piece meshes, child black edge outline line segments, and child pip marker meshes cleanly; pure coordinate, wasteland, piece-slot, and pip placement rules are delegated to `gnostica.board-layout` [12][13][14][15][16][18]. The older CSS-positioned `.board-stage` path remains as a fallback when required Three.js CDN globals are unavailable, the loaded Three.js revision is not r128, or WebGL renderer creation fails, preserving row/column card placement, the matching velvet mat color, faint dotted wasteland outlines, visible color/orientation/pip piece markers, and selected-territory clicks [11][16].

[11] **Three.js CDN Runtime** - `src/main/resources/index.html` loads Three.js plus `examples/js/controls/OrbitControls.js` from pinned `three@0.128.0` CDN URLs before `/js/main.js` with SRI hashes and `crossorigin="anonymous"`, preserving the npm-free shadow-cljs workflow [6]. `gnostica.three-board/runtime-status` only treats the browser runtime as usable when `THREE.REVISION` is `"128"` and `THREE.OrbitControls` exists, otherwise `gnostica.app` falls back to the CSS board with a visible warning [9][15]. `src/main/externs/three.ext.js` declares the global `THREE`, OrbitControls, and the renderer, scene, camera, light, geometry, edge/line, material, mesh, texture loader, and vector APIs used by advanced compilation [9][12][13][16].

[12] **Three.js Board Scene** - `gnostica.three-board/scene` builds a Three.js scene from the existing board cells and shared piece state rather than duplicating game state [8][9][16][18]. It renders nine face-up card planes with existing `/images/*.png` textures over a flat dark velvet table plane, requests WebGL antialiasing and exposes the actual context antialias flag in DOM metadata, uses white card material plus sRGB texture/output encoding so card art is not dimmed, rotates landscape meshes 90 degrees, and spaces centers with the tested `gnostica.board-layout/card-position` rules so orthogonal cards never touch [7][18]. It draws faint dashed line outlines for browser-free `wasteland-spaces` below cards and pieces using r128 `BufferGeometry`, `Line`, and `LineDashedMaterial`, leaving raycasting targeted only at card meshes [14][18]. It also renders Icehouse pieces as lit four-sided cone meshes placed over occupied cards, with size, color, up/cardinal orientation, child black `EdgesGeometry`/`LineSegments` outlines, and child circle pip marker count taken from `gnostica.pieces`; the pip markers use tested side-face descriptors so their disks sit on the front-left triangular pyramid side near the base edge, and raycasting still targets only card meshes [14][16][18]. Lying/cardinal pyramids are rolled 45 degrees around their long axis, tilted by the size-specific side-face correction angle, and lifted by the side-face center height so the contact face lies flush above the card without changing raycast selection [14][16][18]. Texture load failures are logged with card image paths and surfaced through callbacks into `gnostica.app`, while component unmount disposes renderer, control, pointer, geometry, material, texture, and listener resources [11][13][14].

[13] **3D Camera Controls** - `gnostica.three-board/scene` creates `THREE.OrbitControls` against the renderer canvas after setting a z-up camera and default view that keeps the 3x3 board visible [11][12]. [14] **3D Card Selection** - `gnostica.three-board/scene` assigns each card mesh a board index from `src/main/gnostica/board.cljc`, uses `THREE.Raycaster` on click-like pointer releases to pick card planes after camera movement, and invokes the selection callback supplied by `gnostica.app` so the territory panel stays in sync through the existing `::select-board-card` event [8][9][12]. [16] **Icehouse Piece Model and Rendering** - `src/main/gnostica/pieces.cljc` defines six metadata-backed player colors, piece sizes/pips, legal orientations, a small visible demo piece position, helpers for grouping pieces by board space, and square-pyramid geometry helpers for cardinal pieces lying on a side face [6][9][10][12][18]. `gnostica.app-state` injects those demo pieces into the shared game state's `[:pieces :on-board]` slot for the browser viewer, and `gnostica.app` passes that derived board-piece collection into `gnostica.three-board/scene` [9][17]. The CSS fallback renders matching colored triangular/diamond markers with clipped teardrop-style pip marks that rotate with the piece body, also taking over if WebGL renderer creation fails, and the territory panel lists pieces on the selected card [9].

[17] **Game State Contract** - `src/main/gnostica/game_state.cljc` defines the browser-free authoritative gameplay state boundary for tests, Gherkin-style feature steps, and re-frame events [6][10][19]. `create-game` accepts explicit player specs plus either injected `:deck-order` or deterministic `:shuffle-fn`, enforces two to six known player ids backed by complete `gnostica.pieces/players` color/name metadata, rejects malformed or duplicate deck card ids before setup, deals each player a six-card `:hand`, then reuses `gnostica.board/initial-board` for the territory grid and keeps the remaining cards in `:draw-pile` [8][16]. The state value records `:phase`, normalized `:players` with metadata, hand, stash, score/challenge/elimination placeholders, `:players-by-id`, current turn order metadata, `:board`, empty board pieces plus a per-player `[:pieces :stashes]` index derived from player stashes, discard pile, setup/bid hooks, and compact history events [16][19]. `gnostica.app-state/initialize` wraps this result for re-frame by storing successful state under app-db `:game`, preserving explicit setup errors under `:setup-error`, and leaving UI-only fields such as selected board index and staged move selection outside the game contract [9]. Rule helpers such as `advance-turn` and `apply-cup-move` return explicit success/error maps with `:ok?`, `:state`/`:events`, or structured `:error` data; Cup commands require `:player-id`, a territory or hand-card source plus acting piece, a territory or wasteland target, and either orientation or one-point card selection, hand-card sources discard their source card, and app-state confirmation routes completed territory/hand-card selections through `apply-cup-move`, including move-panel wasteland plus one-point-card staging, resetting selection on success and preserving staged selections with structured gameplay errors on rejection [10].
