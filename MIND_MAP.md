# Mind Map Format - Self-Documentation

> **For AI Agents:** This mind map is your primary knowledge index. Read overview nodes [1-5] first, then follow links [N] to find what you need. Always reference node IDs. When you encounter bugs, document your attempts in relevant nodes. When you make changes, update outdated nodes immediately—especially overview nodes since they're your springboard. Add new nodes only for genuinely new concepts. Keep it compact (20-50 nodes typical). The mind map wraps every task: consult it, rely on it, update it. Never change the first 5 numbered points.

[1] **Mind Map Format Overview** - A graph-based documentation format stored as plain text files where each node is a single line containing an ID, title, and inline references [2]. The format leverages LLM familiarity with citation-style references from academic papers, making it natural to generate and edit [3]. It serves as a superset structure that can represent trees, lists, or any graph topology [4], scaling from small projects (<50 nodes) to complex systems (500+ nodes) [5].

[2] **Node Syntax Structure** - Each node follows the format: `[N] **Node Title** - node text with [N] references inlined` [1]. Nodes are line-oriented, allowing line-by-line loading and editing by AI models [3]. The inline reference syntax `[N]` creates bidirectional navigation between concepts, with links embedded naturally within descriptive text rather than as separate metadata [1][4]. This structure is both machine-parseable and human-readable, supporting grep-based lookups for quick node retrieval [3].

[3] **Technical Advantages** - The format enables line-by-line overwriting of nodes without complex parsing [2], making incremental updates efficient for both humans and AI agents [1]. Grep operations allow instant node lookup by ID or keyword without loading the entire file [2]. The text-based storage ensures version control compatibility, diff-friendly editing, and zero tooling dependencies [4]. LLMs generate this format naturally because citation syntax `[N]` mirrors academic paper references they've seen extensively during training [1][5].

[4] **Graph Topology Benefits** - Unlike hierarchical trees or linear lists, the graph structure allows many-to-many relationships between concepts [1]. Any node can reference any other node, creating knowledge clusters around related topics [2][3]. The format accommodates cyclic references for concepts that mutually depend on each other, captures cross-cutting concerns that span multiple subsystems, and supports progressive refinement where nodes are added to densify understanding [5]. This flexibility makes it suitable as a universal knowledge representation format [1].

[5] **Scalability and Usage Patterns** - Small projects typically need fewer than 50 nodes to capture core architecture, data flow, and key implementations [1]. Complex topics or large codebases can scale to 500+ nodes by adding specialized deep-dive nodes for algorithms, optimizations, and subsystems [4]. The methodology includes a bootstrap prompt (linked gist) for generating initial mind maps from existing codebases automatically [1]. Scale is managed through overview nodes [1-5] that serve as navigation hubs, with detail nodes forming clusters around major concepts [3][4]. The format remains navigable at any scale due to inline linking and grep-based search [2][3].

[6] **Gnostica App Overview** - Gnostica is a Clojure/ClojureScript implementation of the tarot-and-Icehouse territory game documented in `docs/rules.txt` [7]. The browser UI is built with Reagent and re-frame in `src/main/gnostica/app.cljs`, with shared pure data helpers in `.cljc` namespaces [8][9][16][17][18]. `src/main/gnostica/app_state.cljc` now initializes the re-frame app-db from the authoritative game state so browser subscriptions do not maintain a second board setup path [9][10][17]. Three.js and OrbitControls are available as pinned, SRI-checked CDN globals gated to r128, and `gnostica.three-board` renders the current board as movable, selectable textured card planes with visible Icehouse pyramid pieces without adding npm [11][12][13][14][15][16][18].

[7] **Rules Source** - `docs/rules.txt` is the current rules reference. The first implemented rule slice is board setup: nine tarot cards are dealt face up into a 3x3 territory grid, and adjacent territory cards alternate between portrait and landscape orientation with no touching edges [8][9].

[8] **Board Model** - `src/main/gnostica/board.cljc` defines the initial 3x3 board. It shuffles the tarot deck before taking nine table cards, records each cell's row, column, `:face :up`, and assigns checkerboard orientation with top-left portrait so every orthogonal neighbor is opposite [7][10]. `initial-board` also accepts an injectable shuffle function so tests can use deterministic deck order [10].

[9] **Board Rendering** - `src/main/gnostica/app.cljs` renders the board as the first screen and owns re-frame subscriptions/events, while app-db initialization and selection helpers live in browser-free `gnostica.app-state`, which stores the `gnostica.game-state/create-game` result under `:game` and derives board cells, on-board pieces, selected territory, and current player from that shared value [8][16][17]. The normal board surface is `gnostica.three-board/scene`, a Reagent lifecycle component that mounts and disposes a Three.js renderer, OrbitControls, raycast picking listeners, card selection highlight meshes, and piece meshes cleanly; pure coordinate and piece-slot rules are delegated to `gnostica.board-layout` [12][13][14][15][16][18]. The older CSS-positioned `.board-stage` path remains as a fallback when required Three.js CDN globals are unavailable, the loaded Three.js revision is not r128, or WebGL renderer creation fails, preserving row/column card placement, visible piece markers, and selected-territory clicks [11][16].

[10] **Test Runner** - `clojure -M:test` now runs `gnostica.test-runner`, which loads the card, board, board-layout, app-state, piece, and game-state tests [8][9][16][17][18]. Card tests verify the deck manifest has 78 unique image files backed by classpath resources, board tests verify there are nine face-up cards, that the board deal uses the injected shuffle before selecting territory cards, and that every horizontal or vertical neighbor has the opposite orientation, board-layout tests verify renderer card coordinates, board-index lookup, visible piece slot limits, compass rotations, and piece height math, app-state tests verify re-frame initialization stores the shared game state under `:game`, surfaces setup errors, derives current player/board/pieces, and updates selected territory through pure helpers, piece tests verify initial rendering data, legal sizes/orientations, board spaces, per-space capacity, and the square-pyramid side-face math used by lying pieces, while game-state tests verify deterministic construction, two-through-six-player metadata, six-card starting hands, full deck accounting across hands/board/draw/discard, deck injection, duplicate/malformed deck rejection, phase/current-player metadata, stash shape, invalid player counts, and explicit transition result maps [7][12][16][17][18]. `clojure -M:smoke` runs `gnostica.smoke-runner`, a npm-free headless Chrome/DevTools browser smoke check for the 3D board against either a local released Ring server or `SMOKE_URL` [15].

[11] **Three.js CDN Runtime** - `src/main/resources/index.html` loads Three.js plus `examples/js/controls/OrbitControls.js` from pinned `three@0.128.0` CDN URLs before `/js/main.js` with SRI hashes and `crossorigin="anonymous"`, preserving the npm-free shadow-cljs workflow [6]. `gnostica.three-board/runtime-status` only treats the browser runtime as usable when `THREE.REVISION` is `"128"` and `THREE.OrbitControls` exists, otherwise `gnostica.app` falls back to the CSS board with a visible warning [9][15]. `src/main/externs/three.ext.js` declares the global `THREE`, OrbitControls, and the renderer, scene, camera, light, geometry, material, mesh, texture loader, and vector APIs used by advanced compilation [9][12][13][16].

[12] **Three.js Board Scene** - `gnostica.three-board/scene` builds a Three.js scene from the existing board cells and shared piece state rather than duplicating game state [8][9][16][18]. It renders nine face-up card planes with existing `/images/*.png` textures, uses white card material plus sRGB texture/output encoding so card art is not dimmed, rotates landscape meshes 90 degrees, and spaces centers with the tested `gnostica.board-layout/card-position` rules so orthogonal cards never touch [7][18]. It also renders Icehouse pieces as lit four-sided cone meshes placed over occupied cards, with size, color, and up/cardinal orientation taken from `gnostica.pieces`; lying/cardinal pyramids are rolled 45 degrees around their long axis, tilted by the size-specific side-face correction angle, and lifted by the side-face center height so the contact face lies flush above the card without changing raycast selection [14][16][18]. Texture load failures are logged with card image paths and surfaced through callbacks into `gnostica.app`, while component unmount disposes renderer, control, pointer, geometry, material, texture, and listener resources [11][13][14].

[13] **3D Camera Controls** - `gnostica.three-board/scene` creates `THREE.OrbitControls` against the renderer canvas after setting a z-up camera and default view that keeps the 3x3 board visible [11][12]. Controls enable mouse, trackpad, and touch orbit/zoom, disable pan to keep the board from drifting away, clamp distance and polar angle to avoid clipping through cards, and expose a `Reset view` button that restores the saved default camera state [9].

[14] **3D Card Selection** - `gnostica.three-board/scene` assigns each card mesh a board index from `src/main/gnostica/board.cljc`, uses `THREE.Raycaster` on click-like pointer releases to pick card planes after camera movement, and invokes the selection callback supplied by `gnostica.app` so the territory panel stays in sync through the existing `::select-board-card` event [8][9][12]. Each card also has a slightly larger hidden highlight plane that follows the card orientation and becomes visible for the selected index; drag distance filtering prevents normal orbit gestures from changing selection [13].

[15] **3D Board Verification** - The CDN-based 3D board path is verified with `clojure -M:test` for pure board, board-layout, and piece invariants, `clojure -M:release` for advanced compilation against `src/main/externs/three.ext.js`, and `clojure -M:smoke` for automated browser coverage [10][11][16][18]. The smoke runner starts the released Ring app unless `SMOKE_URL` targets a running dev/release server, finds Chrome/Chromium via standard paths or `SMOKE_CHROME`, then checks desktop and mobile viewports, r128 Three.js/OrbitControls globals, a nonblank canvas screenshot with visible board content, nine texture loads, absence of happy-path texture/fallback status messages, reset control presence, center-card raycast selection updating the territory panel, and both CSS fallback paths: blocked pinned CDN scripts plus a fake `THREE.REVISION` injected before `/js/main.js` [9][11][12][13][14][16].

[16] **Icehouse Piece Model and Rendering** - `src/main/gnostica/pieces.cljc` defines six metadata-backed player colors, piece sizes/pips, legal orientations, a small visible demo piece position, helpers for grouping pieces by board space, and square-pyramid geometry helpers for cardinal pieces lying on a side face [6][9][10][12][18]. `gnostica.app-state` injects those demo pieces into the shared game state's `[:pieces :on-board]` slot for the browser viewer, and `gnostica.app` passes that derived board-piece collection into `gnostica.three-board/scene` [9][17]. Three.js renders them as `THREE.ConeGeometry` pyramid meshes using `MeshLambertMaterial` plus ambient/directional lights, keeps upright pieces vertical, and lays cardinal pieces flush against card faces while preserving their compass direction with `gnostica.board-layout` height and rotation helpers [12][14][18]. The CSS fallback renders matching colored triangular/diamond markers with pip dots, also taking over if WebGL renderer creation fails, and the territory panel lists pieces on the selected card [9].

[17] **Game State Contract** - `src/main/gnostica/game_state.cljc` defines the browser-free authoritative gameplay state boundary for tests, future Cucumber steps, and re-frame events [6][10]. `create-game` accepts explicit player specs plus either injected `:deck-order` or deterministic `:shuffle-fn`, enforces two to six known player ids backed by complete `gnostica.pieces/players` color/name metadata, rejects malformed or duplicate deck card ids before setup, deals each player a six-card `:hand`, then reuses `gnostica.board/initial-board` for the territory grid and keeps the remaining cards in `:draw-pile` [8][16]. The state value records `:phase`, normalized `:players` with metadata, hand, stash, score/challenge/elimination placeholders, `:players-by-id`, current turn order metadata, `:board`, empty board pieces plus a per-player `[:pieces :stashes]` index derived from player stashes, discard pile, setup/bid hooks, and compact history events [16]. `gnostica.app-state/initialize` wraps this result for re-frame by storing successful state under app-db `:game`, preserving explicit setup errors under `:setup-error`, and leaving UI-only fields such as selected board index outside the game contract [9]. Rule helpers such as `advance-turn` return explicit success/error maps with `:ok?`, `:state`/`:events`, or structured `:error` data instead of depending on browser side effects [10].

[18] **Board Layout Helpers** - `src/main/gnostica/board_layout.cljc` contains browser-free renderer math shared by the Three.js board scene and tests [9][10][12]. It defines card dimensions, territory gaps, table size, board-index lookup, card center coordinates, visible piece slot limits, compass z-rotations, side-face roll, and piece center heights above the card surface [12][16]. Keep deterministic placement math here when it can be checked by `clojure -M:test`; leave DOM, WebGL, texture loading, OrbitControls, and raycasting in `gnostica.three-board` [11][13][14].
