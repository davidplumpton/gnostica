[1] **Mind Map Format Overview** - A graph-based documentation format stored as plain text files where each node is a single line containing an ID, title, and inline references [2]. The format leverages LLM familiarity with citation-style references from academic papers, making it natural to generate and edit [3]. It serves as a superset structure that can represent trees, lists, or any graph topology [4], scaling from small projects (<50 nodes) to complex systems (500+ nodes) [5].

[2] **Node Syntax Structure** - Each node follows the format: `[N] **Node Title** - node text with [N] references inlined` [1]. Nodes are line-oriented, allowing line-by-line loading and editing by AI models [3]. The inline reference syntax `[N]` creates bidirectional navigation between concepts, with links embedded naturally within descriptive text rather than as separate metadata [1][4]. This structure is both machine-parseable and human-readable, supporting grep-based lookups for quick node retrieval [3].

[3] **Technical Advantages** - The format enables line-by-line overwriting of nodes without complex parsing [2], making incremental updates efficient for both humans and AI agents [1]. Grep operations allow instant node lookup by ID or keyword without loading the entire file [2]. The text-based storage ensures version control compatibility, diff-friendly editing, and zero tooling dependencies [4]. LLMs generate this format naturally because citation syntax `[N]` mirrors academic paper references they've seen extensively during training [1][5].

[4] **Graph Topology Benefits** - Unlike hierarchical trees or linear lists, the graph structure allows many-to-many relationships between concepts [1]. Any node can reference any other node, creating knowledge clusters around related topics [2][3]. The format accommodates cyclic references for concepts that mutually depend on each other, captures cross-cutting concerns that span multiple subsystems, and supports progressive refinement where nodes are added to densify understanding [5]. This flexibility makes it suitable as a universal knowledge representation format [1].

[5] **Scalability and Usage Patterns** - Small projects typically need fewer than 50 nodes to capture core architecture, data flow, and key implementations [1]. The methodology includes a bootstrap prompt (linked gist) for generating initial mind maps from existing codebases automatically [1]. Scale is managed through overview nodes [1-5] that serve as navigation hubs, with detail nodes forming clusters around major concepts [3][4]. The format remains navigable at any scale due to inline linking and grep-based search [2][3].

[6] **Gnostica App Overview** - Gnostica is a Clojure/ClojureScript implementation of the tarot-and-Icehouse territory game documented in `docs/rules.txt` [7]. The browser UI is built with Reagent and re-frame in `src/main/gnostica/app.cljs`, while shared `.cljc` namespaces hold board, app-db, game-state, schema, layout, icon, and piece contracts [8][9][16][17][18][19][21]. The current browser surface combines a Three.js board, CSS fallback board, current-player card zones, staged move-selection panel, keyboard command dialog, and special move icon guide [11][12][13][14][15][22][23][24][25][30]. Icon overlays and popup descriptions are shared across board, hand, discard, and smoke paths [30][31].

[7] **Rules Source** - `docs/rules.txt` is the current rules reference. The first implemented rule slice is board setup: nine tarot cards are dealt face up into a 3x3 territory grid, and adjacent territory cards alternate between portrait and landscape orientation with no touching edges [8][9].

[8] **Board Model** - `src/main/gnostica/board.cljc` defines the initial 3x3 board. It shuffles the tarot deck before taking nine table cards, records each cell's row, column, `:face :up`, and assigns checkerboard orientation with top-left portrait so every orthogonal neighbor is opposite [7][10]. `initial-board` also accepts an injectable shuffle function so tests can use deterministic deck order [10].

[9] **Board Rendering** - `src/main/gnostica/app.cljs` renders the board-first screen and owns the re-frame subscriptions/events that connect browser controls to shared app-db helpers [17]. The normal board surface is `gnostica.three-board/scene`, with Three.js runtime gating, camera controls, raycast selection, and browser-free layout math split across focused nodes [11][12][13][14][18][26][27]. The CSS `.board-stage` path remains the fallback board surface and keeps selection behavior aligned with the 3D path [25]. The surrounding UI prioritizes a wider game view, keeps idle side panels compact, derives card zones from `:game`, and stages move choices outside `:game` [23][24][37].

[10] **Test Runner** - `clojure -M:test` runs `gnostica.test-runner`, which loads card, board, board-layout, app-state, piece, game-schema, game-state, and gameplay feature tests [8][16][17][18][19][20][28][29]. Unit coverage is grouped by pure rendering math, card metadata, piece geometry, app-db derivations, game-state transitions, schema invariants, and feature-runner scenarios [20][28][29]. Browser release and smoke coverage for the Three.js board, fallback paths, card zones, and icon mode behavior live outside the unit runner [15][31].

[11] **Three.js CDN Runtime** - `src/main/resources/index.html` loads Three.js plus `examples/js/controls/OrbitControls.js` from pinned `three@0.128.0` CDN URLs before `/js/main.js` with SRI hashes and `crossorigin="anonymous"`, preserving the npm-free shadow-cljs workflow [6]. `gnostica.three-board/runtime-status` requires `THREE.REVISION` `"128"` and `THREE.OrbitControls`, otherwise `gnostica.app` falls back to CSS with a warning [9][25]. `src/main/externs/three.ext.js` declares the global and advanced-compilation APIs used by the 3D renderer [12][13][27].

[12] **Three.js Board Scene** - `gnostica.three-board/scene` is the Reagent lifecycle component that builds the 3D board from shared board cells and piece state instead of duplicating game data [8][9][16][18]. It owns renderer creation, scene/camera setup, OrbitControls wiring, texture callbacks, pointer listeners, and disposal of renderer, controls, geometry, materials, textures, and listeners [11][13][14]. Card planes, wasteland outlines, icon textures, piece meshes, and lifecycle failure reporting are split into renderer detail nodes [26][27][30].

[13] **3D Camera Controls** - `gnostica.three-board/scene` creates `THREE.OrbitControls` against the renderer canvas after setting a z-up camera and default view that keeps the 3x3 board visible [11][12]. Controls enable mouse, trackpad, and touch orbit/zoom, disable pointer pan, clamp distance and polar angle to allow closer card inspection without clipping through pieces, expose zoom clamp plus camera-distance/target metadata for smoke verification, support focused WASD/arrow-key bounded board-plane panning, preserve the active camera distance and target when icon-mode toggling remounts card textures, and expose a `Reset view` button that restores the saved default camera state [9][15][22].

[14] **3D Card Selection** - `gnostica.three-board/scene` assigns each card mesh a board index from `src/main/gnostica/board.cljc`, uses `THREE.Raycaster` on click-like pointer releases to pick card planes after camera movement, and invokes the selection callback supplied by `gnostica.app` so the territory panel stays in sync through the existing `::select-board-card` event [8][9][12]. Each card also has a slightly larger hidden highlight plane that follows the card orientation and becomes visible for the selected index; drag distance filtering prevents normal orbit gestures from changing selection [13].

[15] **3D Board Verification** - The CDN-based 3D board path is verified with `clojure -M:test` for pure board/layout/piece invariants, `clojure -M:release` for advanced compilation against `src/main/externs/three.ext.js`, and `clojure -M:smoke` for automated browser coverage [10][11][16][18]. The smoke runner owns browser checks for Three.js runtime readiness, nonblank rendering, card zones, icon toggles, card selection, and fallback paths [25][31][32].

[16] **Icehouse Piece Model** - `src/main/gnostica/pieces.cljc` defines six metadata-backed player colors, piece sizes/pips, legal orientations, a small visible demo piece position, helpers for grouping pieces by board space, and square-pyramid geometry helpers for cardinal pieces lying on a side face [6][10][18]. `gnostica.app-state` injects demo pieces into `[:pieces :on-board]`, and `gnostica.app` passes the derived board-piece collection into the renderer [9][17]. Three.js and CSS rendering details live in focused nodes [27][33].

[17] **Game State Contract** - `src/main/gnostica/game_state.cljc` is the browser-free authoritative gameplay boundary for tests, Gherkin-style feature steps, and re-frame events [6][10][19]. It owns deterministic game construction, deck accounting, player metadata, turn metadata, piece/stash mirrors, draw/discard piles, setup hooks, and compact history events [16][34]. Rule helpers return explicit success/error maps, with Cup move semantics split into their own transition node [35]. `gnostica.app-state/initialize` stores successful state under app-db `:game` and leaves UI-only fields outside the game contract [9][24].

[18] **Board Layout Helpers** - `src/main/gnostica/board_layout.cljc` contains browser-free renderer math shared by the Three.js board scene and tests [9][10][12]. It defines card dimensions, territory gaps, table size, board-index lookup, card centers, wasteland detection, board-space bounds, outline segments, visible piece slot limits, side-face pip marker descriptors, compass rotations, side-face roll, and piece heights [12][16][26][27]. Keep deterministic placement math here; leave DOM, WebGL, texture loading, OrbitControls, and raycasting in `gnostica.three-board` [11][13][14].
[19] **Gameplay State Schemas** - `src/main/gnostica/game_schema.cljc` adds Malli-backed schemas for pure gameplay data using `metosin/malli` from `deps.edn` [17]. It defines reusable schemas for card references, board cells including post-setup territory growth, six-card hands, card piles, player state, Icehouse pieces including temporary wasteland coordinates, turn metadata, setup data, history events, and the whole `GameState` map [8][16][17]. Whole-state invariant predicates and public validation helpers are split out for scanability [36].
[20] **Gameplay Feature Harness** - `features/` contains Gherkin-style gameplay rule scenarios run by `test/gnostica/gameplay_feature_test.clj` through `gnostica.feature-runner` [10][17][19]. `test/gnostica/feature_world.clj` centralizes deterministic player setup, identity deck order, pure action application, state lookup, deck accounting, and Malli explanations, while `test/gnostica/feature_steps.clj` maps Given/Then text to helpers [17][19]. The setup feature covers two through six players, six-card hands, nine face-up board cards, and complete 78-card accounting; failing steps include scenario, step, summary, and schema details [10].
[21] **Gnostica Icon Stacks** - `src/main/gnostica/icons.cljc` is the browser-free source for Stickas major-arcana icon stacks, icon ids, labels, and the `GnosticaStickas.pdf` reference URL [30]. `src/main/gnostica/cards.cljc` attaches each major card's ordered one-, two-, or three-icon `:gnostica-icons` vector while minors deliberately have none, and it also exposes Cup/one-point predicates used by `gnostica.game-state/apply-cup-move` [17][28]. Layout, rendering modes, hotkey integration, and verification are documented separately so icon-stack updates do not require editing one dense node [22][30][31].
[22] **Keyboard Command Help** - `src/main/gnostica/app_state.cljc` keeps browser-free `:hotkey-help-open?` and `:icon-help-open?` flags plus open/close helpers, while `src/main/gnostica/app.cljs` renders the header controls, modal command list, special move icon guide, and global keydown listener [6][9][21][30]. Supported browser shortcuts are `?` to show keyboard commands, `G` to show detailed special move icon explanations, `I` to toggle card icon overlays without resetting the current 3D OrbitControls view, board-local WASD/arrow keys to move the focused 3D view, and `Esc` to close open help dialogs; keep the visible command list, keyboard listener, app-state tests, smoke checks, README, and AGENTS notes synchronized when adding shortcuts [10][13][15][21].
[23] **Current-Player Card Zones** - `gnostica.app-state/card-zones` derives the current hand, draw pile, discard pile, counts, and top discard card from the authoritative `:game` value rather than duplicating zones at the app-db top level [9][17]. `gnostica.app` renders hand card images/titles, a facedown draw deck affordance with remaining count, and a discard pile empty state or top card [6][9]. These zones are display-only until a pure draw/discard transition exists; Cup hand-card moves already discard their source card through `apply-cup-move` [35].
[24] **Move Selection UI** - `gnostica.app-state` keeps staged move-selection state outside `:game` under `:move-selection`, covering source choice, source territory, hand card, minion, existing target territory, Cup target wasteland, one-point territory card, draw count, orientation, confirmation, cancellation, and structured error/result states [9][17]. Browser board clicks and confirmation behavior are split into the move-flow node [37].
[25] **CSS Board Fallback** - The `.board-stage` CSS board remains usable when required CDN globals are unavailable, the loaded Three.js revision is not r128, or WebGL renderer creation fails [9][11]. It preserves row/column card placement, the matching velvet mat color, faint dotted wasteland outlines, visible color/orientation/pip piece markers, and selected-territory clicks [14][16]. The smoke runner covers blocked CDN scripts and injected revision drift so fallback behavior stays aligned with the 3D path [15].
[26] **Three.js Card and Wasteland Rendering** - `gnostica.three-board/scene` renders nine face-up card planes with `/images/*.png` textures over a dark velvet table plane, requests WebGL antialiasing, exposes actual context antialias metadata, uses white card material plus sRGB texture/output encoding, rotates landscape meshes 90 degrees, and spaces card centers with `gnostica.board-layout/card-position` so orthogonal cards never touch [7][12][15][18]. It draws faint dashed line outlines for browser-free `wasteland-spaces` below cards and pieces using r128 `BufferGeometry`, `Line`, and `LineDashedMaterial`, leaving raycasting targeted only at card meshes [14][18].
[27] **Three.js Piece Meshes** - Three.js renders Icehouse pieces as lit four-sided cone meshes over occupied cards, with size, color, up/cardinal orientation, child black `EdgesGeometry`/`LineSegments` outlines, and child circle pip markers derived from `gnostica.pieces` plus `gnostica.board-layout` side-face descriptors [12][16][18]. Lying/cardinal pyramids are rolled and tilted so the contact face sits flush above the card while preserving compass direction and card raycast selection [14][16][18].
[28] **Core Unit Test Coverage** - Card tests verify the 78-card manifest, classpath image resources, major icon metadata, and Cup/one-point predicates [21]. Board tests verify nine face-up cards, injected shuffle use, and alternating orthogonal orientations [8]. Board-layout and piece tests cover renderer coordinates, wasteland detection, board-space bounds, board-index lookup, visible piece slot limits, pip marker counts/positions, compass rotations, piece height math, legal sizes/orientations, per-space capacity, and lying square-pyramid side-face geometry [16][18].
[29] **Game and App-State Test Coverage** - App-state tests verify initialization under `:game`, setup errors, derived current player/board/pieces/card zones, selected territory updates, move source availability, staged controls, incomplete selections, Cup confirmations, rejected selections without game mutation, and draw/discard derivation without top-level card-zone state [17][23][24][37]. Game-schema and game-state tests cover setup validity, deterministic construction, deck accounting, transition result maps, Cup success paths, and structured rejection cases [17][19][35][36].
[30] **Icon Layout and Rendering Modes** - `src/main/gnostica/icon_layout.cljc` centralizes the shared 2x special-move icon scale, DOM overlay dimensions, Three.js canvas texture dimensions, and fit checks for up to three stacked icons [21][26]. `src/main/gnostica/icon_view.cljs` owns shared inline SVG icon rows so app, fallback, Three.js popovers, and the icon guide use labels from `gnostica.icons` instead of renderer-local text [21][22][25][26]. `gnostica.icons` also owns ordered glossary entries with detailed descriptions for all special move icons, `gnostica.app-state` stores UI-only `:card-icon-mode` plus icon-guide visibility, and `gnostica.app` exposes the Icons toggle, `I` hotkey, icon guide control, and `G` hotkey through the keyboard command dialog [22].
[31] **Icon Verification Paths** - Always mode renders card-face icon stacks; popup mode hides card-art overlays and reveals icon-plus-label descriptions on hover/focus for board, hand, and discard cards [23][30]. `gnostica.three-board` validates and bakes icon stacks into canvas textures in always mode, remounts with plain card textures in popup mode, preserves the active OrbitControls view, and shows a DOM icon description popover for focused or hovered board cards [12][13][26][30]. Test and smoke coverage for icon behavior lives in its own node [39].
[32] **Smoke Runner Coverage** - `gnostica.smoke-runner` starts the released Ring app unless `SMOKE_URL` targets a running server, finds Chrome/Chromium via standard paths or `SMOKE_CHROME`, and checks desktop/mobile viewports, r128 Three.js/OrbitControls globals, velvet table metadata, nonblank screenshots, antialias metadata, nine texture loads, twelve initial wasteland outline targets, visible piece edge outlines, card zones, reset control, WASD/arrow-key camera-target movement, `?` command help, `G` icon-guide help, `I` hotkey view preservation, center-card selection, blocked-CDN fallback, and fake-revision fallback [9][11][12][13][14][15][22][25][30].
[33] **Piece Rendering Surfaces** - Three.js renders pieces as lit `THREE.ConeGeometry` pyramid meshes with `MeshLambertMaterial`, black edge outlines, pale pip markers, upright/cardinal orientation, and board-layout height/rotation helpers [12][16][18][27]. The CSS fallback renders matching colored triangular or diamond markers with clipped pip marks and lists pieces in the selected-territory panel [9][25].
[34] **Game Setup State** - `create-game` accepts explicit player specs plus injected `:deck-order` or deterministic `:shuffle-fn`, enforces two to six known player ids backed by `gnostica.pieces/players`, rejects malformed or duplicate deck cards, deals each player a six-card hand, reuses `gnostica.board/initial-board` for the territory grid, and keeps the remaining cards in `:draw-pile` [8][16][17]. The state records phase, players, `:players-by-id`, turn order, board, pieces/stashes, discard pile, setup/bid hooks, and history [19].
[35] **Cup Move Transition** - `apply-cup-move` returns `{:ok? true :state ... :events [...]}` or structured `{:ok? false :error ...}` maps [17]. Cup commands require `:player-id`, a territory or hand-card source plus acting piece, a territory or wasteland target, and either orientation or one-point card selection [23][24]. Hand-card sources move the source card to discard, wasteland moves create one-point-card territories, and app-state confirmation resets successful selections while preserving rejected ones with error data [29][37].
[36] **Schema Invariants and Helpers** - Whole-game predicates check player counts, unique player ids, `:players-by-id` consistency, turn membership, current-player indexing, piece ownership by participating players, piece references to board cells, stash ownership, board coordinate/orientation consistency, and unique card ids across hands, board, draw pile, and discard pile [17][19]. Public helpers are `valid-game?`, `explain-game`, `assert-valid-game`, and `invariant-errors`; feature steps use readable explanations without coupling to Three.js or re-frame state [20][29].
[37] **Move Selection Flow** - Three.js raycast card picks and CSS fallback card buttons dispatch through the shared board-selection event for inspection plus existing-territory move staging, while Cup wasteland choices are exposed from the move panel [14][24][25]. Completed Cup territory or hand-card confirmations call `apply-cup-move`, update `:game` only on success, and preserve rejected selections with gameplay error data [29][35].
[38] **Game-State Test Details** - Game-schema and game-state tests cover generated two-through-six-player states, invalid player counts, overfull hands, duplicate cards, unknown piece owners, deterministic construction, deck accounting across hands/board/draw/discard, deck injection validation, turn metadata, stash shape, explicit result maps, Cup success paths, hand-source discard behavior, and structured rejection cases [17][19][29][35][36].
[39] **Icon Test Details** - `gnostica.cards-test`, `gnostica.icon-layout-test`, app-state tests, and the smoke runner cover metadata completeness, ordered glossary descriptions, one-/two-/three-icon coverage, blank placeholder rejection, Cup/one-point predicates, icon fit, mode normalization/toggling, `?gnostica-smoke=major-icons`, command help, icon-guide help, view preservation, and popup descriptions across Three.js and CSS fallback paths [10][15][21][22][25][30][31].
