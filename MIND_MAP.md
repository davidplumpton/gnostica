[1] **Mind Map Format Overview** - A graph-based documentation format stored as plain text files where each node is a single line containing an ID, title, and inline references [2]. The format leverages LLM familiarity with citation-style references from academic papers, making it natural to generate and edit [3]. It serves as a superset structure that can represent trees, lists, or any graph topology [4], scaling from small projects (<50 nodes) to complex systems (500+ nodes) [5].

[2] **Node Syntax Structure** - Each node follows the format: `[N] **Node Title** - node text with [N] references inlined` [1]. Nodes are line-oriented, allowing line-by-line loading and editing by AI models [3]. The inline reference syntax `[N]` creates bidirectional navigation between concepts, with links embedded naturally within descriptive text rather than as separate metadata [1][4]. This structure is both machine-parseable and human-readable, supporting grep-based lookups for quick node retrieval [3].

[3] **Technical Advantages** - The format enables line-by-line overwriting of nodes without complex parsing [2], making incremental updates efficient for both humans and AI agents [1]. Grep operations allow instant node lookup by ID or keyword without loading the entire file [2]. The text-based storage ensures version control compatibility, diff-friendly editing, and zero tooling dependencies [4]. LLMs generate this format naturally because citation syntax `[N]` mirrors academic paper references they've seen extensively during training [1][5].

[4] **Graph Topology Benefits** - Unlike hierarchical trees or linear lists, the graph structure allows many-to-many relationships between concepts [1]. Any node can reference any other node, creating knowledge clusters around related topics [2][3]. The format accommodates cyclic references for concepts that mutually depend on each other, captures cross-cutting concerns that span multiple subsystems, and supports progressive refinement where nodes are added to densify understanding [5]. This flexibility makes it suitable as a universal knowledge representation format [1].

[5] **Scalability and Usage Patterns** - Small projects typically need fewer than 50 nodes to capture core architecture, data flow, and key implementations [1]. The methodology includes a bootstrap prompt for generating initial mind maps from existing codebases automatically [1]. Scale is managed through overview nodes [1-5] that serve as navigation hubs, with detail nodes forming clusters around major concepts [3][4]. The format remains navigable at any scale due to inline linking and grep-based search [2][3].

[6] **Gnostica App Overview** - Gnostica is a Clojure/ClojureScript implementation of the tarot-and-Icehouse territory game documented in `docs/rules.txt` [7]. Browser startup and stable re-frame ids live under `src/main/gnostica/app.cljs` and `src/main/gnostica/app/events.cljs`; injected event handlers, deterministic shuffles, fixtures, and focused `src/main/gnostica/ui/*.cljs` namespaces cover lobby, board, overlays, move panel, header, cards, and help [9][22][23][24][44][45]. The browser surface combines local lobby setup, full-window Three.js board with CSS fallback, card/move/territory overlays, score/end-turn/challenge controls, keyboard help, and special move icon guide [11][12][15][22][23][24][25][30][48]. Implemented gameplay includes setup/bids, draw/discard, Cup/Rod/Disc/Sword moves, major powers, World delegation, orient/initial placement, and endgame flow, with browser staging routed through gesture intents or move selection and explicit unavailable results for unwired powers [34][35][37][41][42][43][46][47][50]. Direct manipulation work should start from the checked-in interaction spec [51]. Workflow uses Jujutsu plus `br`: start from `br ready`, inspect with `br show <id>`, and sync tracker changes with `br sync --flush-only`.

[8] **Board Model** - `src/main/gnostica/board.cljc` defines the direct initial 3x3 board helper. When called directly, `initial-board` shuffles the provided deck with its default shuffle or an injected shuffle function before taking nine table cards, records each cell's row, column, `:face :up`, and assigns checkerboard orientation with top-left portrait so every orthogonal neighbor is opposite [7][10]. The authoritative `create-game` setup path orders or shuffles the full deck once, deals player hands first, then calls `initial-board` with `identity` on the remaining board deck so board setup does not double-shuffle or consume cards before hands [17][34].

[9] **Board Rendering** - `src/main/gnostica/ui/board.cljs` renders the board area from the composed `:gnostica.app/board-view` subscription, chooses between `gnostica.three-board/scene` and the CSS fallback, passes renderer callbacks, dispatches stable selection events registered in `src/main/gnostica/app/events.cljs`, and renders an empty pending board state safely before the lobby has created `:game` [11][12][13][14][17][25]. `board-view` caches board cells, pieces, pieces-by-space keyed by `gnostica.pieces/territory-space` or `wasteland-space`, wastelands, board-space bounds, selected index, move legal-target descriptors, icon mode, Three.js runtime status, renderer errors, and derived renderer availability so fallback layout, capability decisions, and target highlighting stay out of the view render body [16][18][24][25][29]. `src/main/gnostica/app.cljs` composes a local lobby screen before game start, then a full-window board shell with closable overlay card zones and side-docked move/territory panels driven by `:open-panels` [6][23][24]. The normal board surface is `gnostica.three-board/scene`, with Three.js runtime gating, camera controls, raycast selection, and browser-free layout math split across focused nodes [11][12][13][14][18][26][27]. The CSS `.board-stage` path remains the fallback board surface with a matching responsive velvet-to-black fade, board-card pieces, wasteland piece markers, selected territory state, and legal/disabled target classes aligned with the 3D path [25]. The surrounding UI lets the board keep the viewport, derives card zones from `:game`, and stages move choices outside `:game` [23][24][37].

[12] **Three.js Board Scene** - `gnostica.three-board/scene` is the public Reagent component that renders the 3D board from shared board cells and piece state instead of duplicating game data [8][9][16][18].
