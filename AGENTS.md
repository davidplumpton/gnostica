# Agent Notes

## Project Overview

Gnostica is a Clojure and ClojureScript tarot app using re-frame and shadow-cljs.

Use `docs/rules.txt` as the local implementation rules reference. The upstream Gnostica PDF remains the provenance source: https://www.looneylabs.com/sites/default/files/pyramid_rules/Gnostica.pdf. If the local text and upstream PDF differ, treat `docs/rules.txt` as authoritative for current repo behavior until the local copy is deliberately reconciled.

## Development Commands

```sh
clojure -M:dev       # start the ClojureScript watcher
clojure -M:release   # compile an optimized browser build
clojure -M:server    # serve existing released assets with Clojure/Ring
clojure -M:test      # run Clojure tests and gameplay feature scenarios
clojure -M:smoke     # run the headless Chrome 3D board smoke check
```

The shadow-cljs dev server serves the app at `http://localhost:8080/index.html`. The default workflow does not use npm.

Run `clojure -M:release` before `clojure -M:server` or the default `clojure -M:smoke` path. Both serve the existing browser bundle from `src/main/resources/js/main.js`; neither command rebuilds stale or missing JavaScript assets. `SMOKE_URL` can point smoke at an already-running dev or release server instead.

## Shared Game State

`src/main/gnostica/game_state.cljc` defines the stable browser-free gameplay facade. Use `gnostica.game-state/create-game` when tests, Gherkin-style feature steps, or re-frame events need an authoritative state value with deterministic deck setup and explicit success/error results. Setup and shared state helpers live in `src/main/gnostica/game_state/core.cljc`; keep the public facade stable when moving internals. It enforces two to six known players backed by `gnostica.pieces/players` metadata, accepts only official target scores 8, 9, or 10, applies `:deck-order` or `:shuffle-fn` to the full source deck once, deals six-card starting hands before the territory board, then calls `gnostica.board/initial-board` with `identity` on the remaining cards so normal game setup does not reshuffle the board slice. It validates injected deck cards for unique ids plus rendering fields, stores phase, players, turn order, pieces/stashes, draw/discard piles, setup/bid metadata, winner state, and history events, and can apply official `:starting-bids` or `gnostica.game-state/apply-starting-bids` by ranking major bids above minors, rebidding tied top minors, redrawing bid cards counterclockwise from the winner's right, and rotating the winner into the first turn. Use `gnostica.game-state/with-board-pieces` when seeding demo or test pieces so player stashes and `[:pieces :stashes]` stay reconciled with active pieces.

`gnostica.game-state/scores`, `player-score`, and `with-current-scores` derive score from controlled territories only: a territory counts once when occupied by one player's pieces and is worth 1, 2, or 3 for spot, royalty, or major cards. Successful public gameplay transitions refresh stored player scores through the core success helper. `gnostica.game-state/end-turn` is the public facade for challenge and endgame flow: it requires the current non-eliminated player, advances to the next non-eliminated player, optionally announces a final-turn challenge when no other challenge is unresolved, resolves that player's next completed turn as either a target-score win or elimination, discards an eliminated player's hand, removes their pieces while reconciling stashes, records a winner when the challenger succeeds or only one active player remains, and rejects turn advancement after `:finished`.

`gnostica.game-state/apply-cup-move` is the public facade for the pure Cup transition implemented in `src/main/gnostica/game_state/cup.cljc`. Keep its command contract explicit: `:player-id`, a `:source` map for either `:territory` or `:hand-card` plus acting `:piece-id`, carried `:cup-variant`, and a `:target` map for `:territory`, `:piece`, or `:wasteland`. Territory targets require `:orientation` and create the current player's small piece; enemy piece targets preserve the target piece orientation and create one of that enemy's small pieces; wasteland targets use `:territory-card-source :hand` with `:one-point-card-id` or Wheel-only `:territory-card-source :draw-pile-top`. Plain Cup and Magician wild-suit Cup enforce the normal three-piece target limit, Empress-style `:cup-unbounded` ignores that limit, and draw-pile territory creation is rejected for non-Wheel variants. It should continue to return structured success/error maps, update player and stash mirrors together, move hand-source cards to discard, create wasteland territories without duplicating cards, append history events, and keep renderer/UI concerns out of the transition namespace.

`gnostica.game-state/apply-rod-move` is the public facade for the pure Rod transition implemented in `src/main/gnostica/game_state/rod.cljc`. Keep it layered on `resolve-rod-command`: support `:move-minion`, `:push-piece`, and `:push-territory`; carry the resolved `:rod-variant` from plain Rod/Wands, Magician wild suits, or Emperor-style `:rod-unbounded` sources; move hand-source Rod cards to discard; update piece locations as either `:space-index` territories or `:space {:kind :wasteland ...}`; enforce non-void piece destinations; enforce the three-piece territory limit for plain and wild-suit Rod piece destinations while allowing `:rod-unbounded` to enter full territories; allow orientation only for current-player moved pieces; preserve enemy orientation; reject territory pushes when the target territory or landing wasteland has enemy pieces; leave pieces at the source coordinate when a territory moves; place moved territories under own pieces already in the landing wasteland; update pushed territory orientation from the board pattern; and append history events.

`gnostica.game-state/resolve-disc-command` is the public facade for browser-free Coins/Discs command validation implemented in `src/main/gnostica/game_state/disc.cljc`, and `gnostica.game-state/apply-disc-move` applies resolved piece-growth and territory-growth targets. `gnostica.game-state/resolve-major-source`, `charge-major-source-once`, `major-paid-source-opts`, `major-action-source`, and `apply-major-sequence` are the shared browser-free major arcana sequencing foundation implemented in `src/main/gnostica/game_state/major.cljc`. ## Browser JavaScript Globals

`src/main/resources/index.html` loads Three.js and OrbitControls from pinned `three@0.128.0` CDN URLs before `/js/main.js`. The script tags carry SRI hashes and `crossorigin="anonymous"`. This keeps the default workflow npm-free while exposing the browser globals `THREE` and `THREE.OrbitControls` to `gnostica.three-board`, which only enables the 3D renderer when `THREE.REVISION` is `"128"` and OrbitControls is present.

`src/main/gnostica/app/events.cljs` captures Three.js runtime capability through the re-frame initialization coeffect boundary and stores it under app-db `:three-runtime-status`. Keep CSS fallback decisions in `:gnostica.app/board-view` subscription data instead of reading browser globals from view render paths. Renderer lifecycle code may still guard WebGL setup defensively before mounting Three.js resources.

`src/main/externs/three.ext.js` declares the current `THREE` global, OrbitControls, and the Three.js APIs used by the app for advanced compilation, including the small circle geometries used for pyramid pip markers and the edge/line APIs used for black pyramid outlines. Keep any future Three.js add-ons on the same `three@0.128.0` release line and use CDN scripts compatible with the global build. If the CDN URLs change, update the SRI hashes, the runtime revision gate in `src/main/gnostica/three_board/runtime.cljs`, and the smoke checks together.

## Version Control: Jujutsu

This project uses Jujutsu (`jj`) as its VCS. Prefer Jujutsu commands when inspecting or recording work:

```sh
jj status
jj diff
jj describe -m "Describe the current change"
jj commit -m "Commit message"
jj new
```

When starting work, set a Jujutsu change description with `jj describe -m "..."` before editing. At the end of the work, run `jj new` so the completed described change is left behind and the working copy moves to a fresh change.

Do not use git-only workflows unless the user asks for them or a tool requires them.
