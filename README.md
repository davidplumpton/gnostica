# Gnostica

A Clojure and ClojureScript tarot app using re-frame and shadow-cljs. The current app is a local browser implementation focused on local lobby setup, full-window board rendering, shared pure game state, and staged plus ribbon-assisted browser move selection for the implemented gameplay powers. `gnostica.app` is the small browser shell/startup entry point; public re-frame ids are exposed through `gnostica.app.events`, complete id metadata lives in `gnostica.app.ids`, focused registration side effects live under `gnostica.app.registrations.*`, keyboard startup lives under `gnostica.app.*`, fixture/demo init options live in `gnostica.fixtures`, and focused view namespaces under `gnostica.ui.*` render the lobby, board fallback, overlay card zones, territory panel, move panel, header, and help dialogs. The Ring server only serves the released app assets; multiplayer sync and state import/export workflows are not implemented yet.

The implementation rules reference is `docs/rules.txt`, the local text copy used while building and testing gameplay behavior. The upstream Gnostica PDF remains the provenance source: https://www.looneylabs.com/sites/default/files/pyramid_rules/Gnostica.pdf. If the local text and upstream PDF differ, treat `docs/rules.txt` as authoritative for this repo's current implementation context until the local copy is deliberately reconciled.

The game was designed by John Cooper, Kory Heath, Kristin Matherly, and Jacob Davenport. It makes use of a deck of Tarot cards and Looney Pyramids (https://www.looneylabs.com/pyramids-home).

## Development

Install Java, the Clojure CLI, and Node.js, then run:

```sh
clojure -M:dev
```

The shadow-cljs dev server serves the app at `http://localhost:8080/index.html`. Node.js is still required by shadow-cljs for browser builds, but the default workflow does not use npm.

For the released Ring server path, run `clojure -M:release` first. `clojure -M:server` serves the existing files under `src/main/resources`, including the compiled browser bundle at `src/main/resources/js/main.js`; it does not rebuild stale or missing JavaScript assets.

Run `clojure -M:lint` to verify the br tracker guard, clj-kondo warning/error findings, and cljfmt formatting. The guard fails when `.beads/embeddeddolt` is missing, is a directory left by legacy `bd`, is not a regular file, or when `.beads/issues.jsonl` is absent. The clj-kondo and cljfmt checks cover `src/main`, `test`, `deps.edn`, and `shadow-cljs.edn`; the generated browser bundle under `src/main/resources/js` is excluded from these source checks.

The browser runtime loads Three.js and OrbitControls from pinned `three@0.128.0` CDN scripts with SRI hashes and `crossorigin="anonymous"` before the compiled ClojureScript bundle. This keeps browser builds npm-free while still exposing the global `THREE` and `THREE.OrbitControls` values used by the `gnostica.three-board` renderer. The re-frame initialization boundary captures the runtime capability status into app-db, and the board view-model enables the 3D renderer only when `THREE.REVISION` is `"128"` and OrbitControls is present.

## 3D Board View

The normal browser first screen is the local lobby. After the lobby starts a game, or when fixture query options intentionally bypass the lobby for smoke/dev scenarios, the app renders the nine-card territory board with Three.js when the CDN globals are available. The public renderer component is `gnostica.three-board/scene`; focused `gnostica.three-board.*` namespaces handle runtime detection, lifecycle orchestration, resource cleanup, camera controls, pointer picking, and scene graph assembly. It uses the shared board and Icehouse piece models, loads tarot card images as textures through `gnostica.three-card-textures`, alternates portrait and landscape card planes over a dark velvet table surface that fades to black beyond the wastelands, leaves small gaps between neighboring cards, and uses `OrbitControls` for mouse, trackpad, and touch camera movement. The board canvas fills the available browser window below the sticky header, while cards, move controls, and territory details open as docked overlays above it. The focused 3D board also supports WASD and arrow-key panning with bounded camera-target movement so the board remains recoverable. Renderer remounts caused by icon-mode changes, board cell or piece updates, and direct-manipulation option changes restore the current OrbitControls target and distance; only the `Reset view` control intentionally returns the camera to its default view. Browser-free card coordinate, wasteland detection, table fade sizing, board-index, overflow-aware piece slot, pip marker, and piece height helpers live in `gnostica.board-layout` and are covered by Clojure tests. Card picking is handled with a Three.js raycaster and updates the selected territory panel through callbacks from `gnostica.ui.board` into the stable `:gnostica.app/select-board-card` event.

Wasteland spaces are the empty orthogonal neighbors of current territory cards. The Three.js board draws faint dashed outlines for those spaces below the cards and pieces, and the CSS fallback renders a matching velvet-to-black play mat with non-interactive dotted outlines behind the card buttons so legal empty spaces remain visible without affecting selection.

Icehouse pieces are represented in shared state by `gnostica.pieces`. Normal local browser startup does not seed demo pieces; visible demo and smoke pieces live in `gnostica.fixtures` and are applied only when explicit init options provide `:demo-board-pieces`. Those fixture-backed pieces let the renderer display player color, size/pips, up/cardinal orientation, and up to three pieces on one territory without making fixtures part of the production piece model. Three.js renders these as lit four-sided cone meshes over the cards with black edge outlines and one, two, or three pale pip markers near the base; the CSS fallback renders matching color/orientation/pip markers and remains clickable.

If either CDN global is missing, if the loaded Three.js revision is not r128, or if the browser cannot create a WebGL renderer, the app falls back to the CSS board so the game remains usable while surfacing a runtime warning from subscription data. Advanced compilation support for the CDN global is declared in `src/main/externs/three.ext.js`, including the geometry and line APIs used for piece edge outlines; keep future Three.js add-ons on the same `three@0.128.0` release line unless the CDN scripts, SRI hashes, revision gate, smoke checks, and externs are updated together.

## Shared Game State

The authoritative gameplay state contract starts at the `gnostica.game-state` facade. It is browser-free and stable for `clojure.test`, the custom Gherkin-style feature harness, and re-frame events. Focused implementation code lives under `gnostica.game-state.*`: `core` owns setup, shared state, scoring, turns, and endgame helpers; `draw`, `cup`, `rod`, `disc`, `sword`, and `placement` own the single-family transitions; `major` and `major-power` provide reusable major arcana sequencing and delegation support; `composite`, `manipulation`, and `world` own the concrete major-power families; `spatial` and `card-source` hold shared targeting and card lookup helpers; and `command-contracts` owns Malli command and result shapes. `create-game` accepts explicit player specs plus either an injected `:deck-order` or deterministic `:shuffle-fn`, accepts the official 8, 9, or 10 point target score, enforces two to six known players, deals each player a six-card starting hand before the territory board, reuses `gnostica.board` for the nine-card territory grid, can resolve official starting bids from `:starting-bids`, and models phase, players, turn order, pieces/stashes, draw pile, discard pile, setup/bid metadata, winner state, and compact history events. `:deck-order` and `:shuffle-fn` order the full source deck once at the `create-game` boundary; after hands are dealt, `create-game` passes the remaining cards to `gnostica.board/initial-board` with `identity` so normal setup does not reshuffle the board cards.

Player ids must reference the six metadata-backed entries in `gnostica.pieces/players` so every legal setup has a display name, Three.js color, and CSS color. Injected decks are validated before board setup: cards must be ordered maps with unique string `:id` values plus nonblank `:title` and `:image` fields needed by rendering and territory details.

Starting deck accounting is explicit: the first six cards per player become player hands, the next nine cards become board territories, the remaining cards become `:draw-pile`, and `:discard-pile` starts empty. Setup records the ordered expected deck ids under `[:setup :deck-card-ids]`, so schema validation can account for the official 78-card tarot deck and for injected test decks using the same contract. Optional `:starting-bids` remove bid cards from player hands for one or more rounds, resolve the official major/minor ordering including ties, redraw all bid cards from the revealed pool counterclockwise from the winner's right, and rotate turn order so the winner starts. Each player map also carries its initial Icehouse `:stash`, mirrored by the existing `[:pieces :stashes]` index. Tests assert that every expected deck card appears exactly once across those zones.

The browser app initializes through `gnostica.app-state/initialize`, which can either create `:game` immediately or start in a top-level `:lobby` state before any game exists. `gnostica.app-state` is the stable public facade; focused implementation code lives under `gnostica.app-state.db`, `.lobby`, `.view-models`, `.gestures`, and `.moves`. The local lobby lets two to six players join, edit display names, choose unique player colours from `gnostica.pieces/players`, and choose the target score. It can start with the first lobby seat as a casual shortcut, or stage the official first-player bid: `:gnostica.app/start-lobby-bidding` creates an uncommitted initial game so each player can choose a bid card from their six-card hand, `gnostica.game-state/resolve-starting-bid-rounds` previews revealed bids and tied rebid rounds through the shared ranking rules, the lobby then stages bid-card redraw choices in official redraw order from the remaining revealed bid-card pool, and `:gnostica.app/confirm-lobby-bidding` finalizes through `gnostica.game-state/apply-starting-bids` so chosen redraws, six-card hands, and winner-first turn rotation stay in the pure game-state contract. `gnostica.app.ids` owns the stable `:gnostica.app/...` keyword ids, `gnostica.app.events` re-exports the public event and composed view subscription ids consumed by the UI, and `gnostica.app.registrations.events` plus `.subscriptions` install the re-frame registrations. `gnostica.app.handlers` keeps initialization, lobby start, lobby bidding, and move-confirmation transitions testable with injected shuffle seeds. Browser events inject a seed at the re-frame cofx boundary and convert it through `gnostica.deterministic-shuffle`, so lobby setup and draw-pile reshuffles do not rely on implicit randomness inside `reg-event-db` handlers. Board, piece, selected-territory, current-player, score, challenge, and winner subscriptions read through the shared game value instead of maintaining a second top-level board or piece setup path, overlay visibility stays in top-level `:open-panels`, and browser views consume composed feature-level view-model subscriptions for the lobby, board, card zones, territory panel, move panel, pending move tray, header, app shell, and help dialogs. `app-state/initialize` applies visible demo pieces only when explicit `:demo-board-pieces` options are provided; `gnostica.fixtures/browser-init-options` starts the normal local path in a prefilled lobby, while `?gnostica-dev-shared-control=1` tags those lobby seats with one shared local controller for dev testing, `?gnostica-dev-demo-hotkeys=1` enables dev-only browser hotkeys, and `?gnostica-smoke=...` modes bypass the lobby for smoke fixtures. `?gnostica-smoke=major-icons` uses all six players, the smoke board pieces including the wasteland piece, and the major-icon deck order; `?gnostica-smoke=direct-drop` keeps the default two players, opens the cards and move panels, seeds no demo pieces, and uses identity shuffle for first-placement/direct-drop confirmation flows. Lobby controller metadata is stripped before game creation. The pieces are kept in the game state's `[:pieces :on-board]` slot through `gnostica.game-state/with-board-pieces`, so renderer smoke checks that opt into fixture pieces still exercise piece layout while player stashes and the `[:pieces :stashes]` mirror are debited consistently.

The main screen also displays the current player's hand, the draw deck, and the discard pile from `:game` in a closable bottom overlay. The hand zone shows the current player's card images and titles, the draw deck shows a facedown deck affordance with remaining-card count, and the discard pile shows an empty state until a top discarded card exists. These zones derive from pure gameplay state: the standalone draw/discard turn option uses `apply-draw-move`, draw-focused major powers use `apply-fool-move`, `apply-high-priestess-move`, or `apply-judgement-move`, while Cup, Rod, Disc, Sword, Hierophant, Hermit, and Devil hand-card move transitions move the source card from hand to discard.

Rule helpers return explicit result maps: successful transitions use `{:ok? true :state ... :events [...]}`, while validation errors use `{:ok? false :error {:code ... :message ... :data ...}}`. Public turn-action facades share a common guard for unknown, non-current, eliminated, and finished players, and no-piece players must place their initial small piece before drawing, announcing a challenge, ending the turn, or taking another non-placement turn action. Successful transitions refresh player scores from territory control: a territory scores only when occupied by one player's pieces, with spot, royalty, and major territories worth 1, 2, and 3. `gnostica.game-state/end-turn` advances to the next non-eliminated player, optionally announces the current player's final-turn challenge, resolves an existing challenge after that player's next turn as either a win or elimination, discards an eliminated player's hand, removes their pieces, and records a last-active-player win when only one player remains. The public `gnostica.game-state/apply-draw-move` facade delegates to `gnostica.game-state.draw`; it accepts `:player-id`, optional `:discard-card-ids`, `:draw-count`, and an optional deterministic `:shuffle-fn`, requires the current player to already have a piece on the board, discards selected hand cards, draws toward the six-card hand limit, reshuffles the discard pile when the draw pile is exhausted, preserves deck accounting, and appends history.

The same draw namespace owns browser-free draw/discard major powers. `apply-fool-move` charges a Fool hand source once, reveals up to two draw-pile cards into discard, and can route an optionally played revealed card through currently implemented suit and full-card major transitions without charging that revealed source again. `apply-high-priestess-move` charges High Priestess once and applies up to two sequential discard/redraw passes. `apply-judgement-move` charges Judgement once, validates the active minion, and draws selected discard-pile cards into hand up to that minion's pip count and the six-card hand limit; hand-source Judgement can draw itself back after source cost.

`gnostica.game-state/apply-cup-move` delegates to `gnostica.game-state.cup` and accepts an explicit command with `:player-id`, a source map for either a Cup territory or Cup hand card plus acting `:piece-id`, a carried `:cup-variant`, and a target map for an existing territory, enemy piece, or wasteland coordinate. It places the current player's small piece with `:orientation`, targets enemy pieces to create one of that enemy's small pieces with the target's orientation, creates territories only in the acting minion's targeted wasteland from hand one-point cards or Wheel's optional top draw-pile card, and accepts optional `:shuffle-fn` for deterministic discard reshuffles when Wheel exhausts the draw pile. It discards hand-source Cup cards, preserves deck and schema invariants, and reports structured rejection errors. Plain Cup and Magician wild-suit Cup use the normal three-piece limit, Empress-style `:cup-unbounded` ignores that target-space limit, and Wheel-only draw-pile territory creation is rejected for other Cup variants.

`gnostica.game-state/apply-rod-move` delegates to `gnostica.game-state.rod` and builds on `resolve-rod-command` for pure Rod movement. It supports moving the acting minion, pushing a target piece, and pushing a target territory in the minion's facing direction, while carrying the resolved `:rod-variant` from plain Wands/Rod, Magician wild suits, or Emperor-style `:rod-unbounded` sources. Piece moves enforce positive distance up to the minion pip count, reject upright minions and void destinations, reject full destination territories for plain and wild-suit Rods, allow `:rod-unbounded` moves into full territories, represent wasteland destinations with explicit `:space` coordinates, allow orientation changes only for moved current-player pieces, and preserve enemy orientations. Territory pushes reject enemy-occupied target territories and enemy-occupied landing wastelands, land only in wasteland spaces, leave pieces behind at the source coordinate, place the moved territory under own pieces already in the landing wasteland, update the territory orientation for the board pattern, discard hand-source Rod cards, and append history events.

`gnostica.game-state/resolve-disc-command` delegates to `gnostica.game-state.disc` for browser-free Coins/Discs command validation, and `apply-disc-move` consumes resolved piece and territory targets. Commands accept `:player-id`, a territory or hand-card source plus acting `:piece-id`, a carried `:disc-variant`, a target piece or territory, optional replacement-card source/id for territory growth, and optional orientation for current-player piece growth. Targeting follows upright current-space, cardinal adjacent-space, and minion self-targeting rules. Piece growth swaps small pieces to medium and medium pieces to large when the target owner's larger stash piece is available, returns the old piece to that stash, preserves enemy orientation, optionally reorients current-player pieces, discards hand-source Disc cards, appends history, and keeps schema/card/stash accounting valid. Territory growth replaces a legal target territory with a hand or Star discard-pile card worth exactly one more point, moves the old territory to discard, preserves pieces on that coordinate with their orientations intact, appends history, and keeps unique card accounting valid. Strength hand or territory sources can pass `:disc-actions` with one or two action maps; when both actions target the same piece or territory, the transition can skip the intermediate piece size or territory value. Star can pass `:minion-orientation` so the orient step resolves before the Disc target is validated, while still allowing a just-discarded hand-source Star to become a discard-pile territory replacement. `gnostica.game-state/apply-sun-move` applies Sun's Cup-then-Disc order with `:cup` and optional `:disc` subcommands; same-created-target shortcuts can create a medium piece without a small stash piece or create a royalty territory without spending a one-point card. Browser move selection exposes Sun as its own staged Cup-then-optional-Disc power, including created-piece and created-territory shortcuts, while keeping the card's separate Cup and Disc powers selectable.

`gnostica.game-state/resolve-major-source` and `apply-major-sequence` provide shared browser-free scaffolding for ordered, optional multi-power major arcana turns. The engine validates major hand-card and territory sources, charges a hand source once, starts hand-play with all owned pieces as minions, starts territory activation with owned pieces touching that card, enforces fixed power order as a subsequence, promotes affected owned pieces into later-step minions, and exposes same-target shortcut hooks for composite powers.

`gnostica.game-state/apply-empress-move`, `apply-emperor-move`, `apply-lovers-move`, `apply-chariot-move`, `apply-hanged-man-move`, and `apply-temperance-move` delegate to `gnostica.game-state.composite` for browser-free composite suit-major powers. Commands carry `:player-id`, a major `:source`, and optional ordered `:actions`: Empress orients then uses unbounded Cup, Emperor orients then uses unbounded Rod, Lovers applies Rod then Cup, Chariot applies Rod twice with a same-piece shortcut that may pass through full territories or void but cannot end there, Hanged Man can trade hands only or apply Rod then trade hands with the owner of a targeted piece, and Temperance applies Cup twice. Hand sources charge once, territory sources keep major-sequence minion promotion, and the underlying Cup/Rod transitions keep card, stash, and schema accounting intact. Browser move selection exposes these as explicit ordered major powers, collecting completed prerequisite actions under `:major-actions`, showing done/active/skipped steps in the action ribbon, and confirming through the matching composite facade.

`gnostica.game-state/resolve-sword-command` delegates to `gnostica.game-state.sword` for browser-free Swords command validation, and `apply-sword-move` consumes resolved piece and territory targets. Single Sword commands accept `:player-id`, a territory or hand-card source plus acting `:piece-id`, a carried `:sword-variant`, a target piece or territory, positive `:damage`, optional replacement-card source/id for non-destroying territory attacks, and optional orientation only when shrinking a surviving current-player piece. The resolver identifies Swords minors, Magician wild suits, plain Sword majors such as Justice/Death/Moon, and Tower's discard-pile replacement variant; it returns structured success/error maps without mutating state. Piece attacks shrink surviving victims to the appropriate smaller stash piece, destroy zero-pip victims, preserve enemy survivor orientation, optionally reorient current-player survivors after resolver validation, discard hand-source Sword cards, append history, and keep schema/card/stash accounting valid. Territory attacks replace surviving territories with a hand or Tower discard-pile card worth the original value minus the selected damage, reject missing, reused, or invalid replacements, destroy territories when damage equals their value, discard removed territory cards, return any pieces left in void to their owner stashes, append history, and keep schema/card accounting valid. Sword-related major composites are also browser-free: Justice commands can pass `:hand-trade-target` or `:hand-trade-target-piece-id` to charge the source once and either swap hands with the target piece owner only or swap before applying Sword; Death commands can pass one or two `:sword-actions`, with same-target shortcuts that skip missing intermediate piece sizes or territory values; Tower commands can pass `:minion-orientation` before Sword; and `gnostica.game-state/apply-moon-move` applies Moon's optional Rod then optional Sword sequence, including the temporary full-territory Rod entry that must be resolved by the Sword action. Browser move selection exposes both the plain Sword subset and dedicated Justice, Death, Tower, and Moon staged powers: Justice stages trade-only or hand trade then Sword, Death stages one or two Sword actions, Tower stages minion orientation then Sword with discard-pile territory replacements, and Moon stages Rod then Sword.

`gnostica.game-state/apply-hierophant-move`, `apply-hermit-move`, and `apply-devil-move` delegate to `gnostica.game-state.manipulation` for browser-free non-suit major powers. These transitions use the shared major-source sequence rules: hand sources make all owned pieces minions and charge the card once; territory sources use owned pieces on the source card; owned affected pieces can become later minions. Hierophant replaces a target piece with the current player's same-size stash piece and orients it. Hermit moves a targeted piece to an empty territory or wasteland, or moves a targeted territory to an eligible wasteland while leaving source-space pieces behind. Devil applies up to three target-piece orientations, including enemy pieces, with each action retargeted against the current post-orientation state.

`gnostica.game-state/world-major-territories` enumerates non-World major arcana territories that World can copy, and `apply-world-move` applies the chosen copied power while using World as the paid source. Commands carry `:player-id`, a World `:source`, `:copied-board-index`, optional `:copied-power` or legacy `:power` for the selected suit or implemented full-major power, plus the copied card's normal command fields. World delegates to the existing draw, suit, composite, manipulation, Sun, and Sword-major implementations with separate paid-source and copied-power cards so source cost, minion scope, variants, history source summaries, and schema/card accounting stay aligned. Browser move selection exposes World by choosing a copyable major territory with its stable board index, choosing the copied card power when needed, showing the copied sequence in the action ribbon, reusing the copied power's detailed child controls, and confirming through `apply-world-move`.

`gnostica.game-state/apply-orient-move` and `gnostica.game-state/apply-initial-placement` delegate to `gnostica.game-state.placement`. Orient turns one current-player piece to a legal orientation, preserving the same board or wasteland space and appending history. Initial placement is the special no-piece turn rule: when the current player has no pieces on the board, they may place their first small piece onto an empty territory or empty wasteland in the selected orientation while keeping player and stash mirrors reconciled.

## Gameplay Move Selection

The browser app includes a current-player move panel rendered by the public `gnostica.ui.move-panel` facade and driven from shared app-db state in `gnostica.app-state`; focused renderer namespaces under `gnostica.ui.move-panel.*` own source picking, reusable choices, power control groups, the action ribbon, the pending tray, and the shell. Move selection is stored outside `:game` under `:move-selection`, gesture-started pending moves are stored under `:gesture-intent`, the current turn's consumed move is tracked under top-level `:turn-action`, direct-manipulation session settings live under top-level `:direct-manipulation`, and panel visibility is stored under `:open-panels`; board, pieces, card zones, hand cards, draw counts, player metadata, scores, target score, active challenges, and winner state continue to derive from the authoritative game state. The staged move-selection facade lives in browser-free `gnostica.move-selection`, with `gnostica.move-selection.registry` centralizing move source and power labels, card-to-power mapping, control kinds, command builder keys, transition facades, previewability, and renderer control keys; focused `.options`, `.controls`, `.targets`, `.commands`, `.confirmation`, `.ribbon`, and `.preview` namespaces own option/prompt metadata, option/control groups, legal-target descriptors, command construction, transition dispatch and app-db result application, action-ribbon view models, and board preview view models behind the stable public API. `gnostica.app-state.moves` re-exports the move-selection facade through the public app-db namespace and owns confirmation/end-turn app-db wiring. `gnostica.app-state.gestures` bridges direct gestures to the staged state, while `gnostica.gesture-intent` lets hand-card, territory-card, draw-pile, piece, stash-piece, and replacement-card gestures populate the same staged fields and pending tray before confirmation. `docs/direct_move_entry_spec.md` is the active interaction contract for direct manipulation, defining the object grammar, visual states, ambiguity rules, Detailed entry fallback path, and command-shape examples that gesture implementations, tests, and smoke coverage follow. Detailed entry can be made the session default, opening the move panel from setup and from pending gestures without clearing valid staged choices; pointer dragging can also be disabled while classic staged controls remain available. `gnostica.app-state` remains the app-db boundary and exposes the UI-facing facade used by re-frame subscriptions and events. `:gnostica.app/move-panel-view` composes the current selection, prompt, ready state, selected power, action ribbon, board cells, direct-manipulation settings, and all option lists so the detailed panel renders from one feature-level subscription. `:gnostica.app/pending-move-tray-view` is a separate public composed subscription for gesture-started pending moves; the app shell uses it to decide when to show the side stack, and `gnostica.ui.move-panel.pending-tray` renders its Confirm, Cancel, Detailed entry, missing-choice, error, and action-ribbon state directly from that boundary. The header exposes score chips plus end-turn and challenge controls that call the pure `end-turn` transition and stay disabled until a no-piece current player completes initial placement.

Move powers without browser confirmation wiring return an explicit `:move-transition-unavailable` result instead of mutating game state.

## Keyboard Shortcuts

The app header in `gnostica.ui.header` exposes panel toggles plus a `?` control, `gnostica.keyboard-shortcuts` owns the shared shortcut command catalog, `gnostica.app.keyboard` installs the global browser listener through a re-frame effect boundary, and `gnostica.ui.help` renders the command and icon dialogs from the same catalog. Help dialogs are real modals: opening one focuses its close control, traps Tab/Shift+Tab inside the dialog, marks the header/game chrome inert and `aria-hidden`, restores the previous focus on close, and contains global shortcuts other than Escape. The dialog lists the currently supported browser shortcuts: `?` opens the dialog, `G` opens a special move icon guide with detailed explanations, `I` toggles Gnostica card icon overlays between always-visible and hover/focus popup modes, WASD or arrow keys move the focused 3D board view when no orientation-capable drag or keyboard first-placement targeting is active, Enter or Space on the Place first piece source starts keyboard target selection, arrow keys move that target among legal empty territories and wastelands, Enter or Space accepts the target and later confirms once ready, `O` cycles the orientation for an eligible dragged piece or pending first-piece placement, and `Esc` closes open help dialogs or cancels active keyboard placement targeting. While a piece or first-placement stash piece drag is active, arrow keys can set the pending orientation before drop: up, east, south, or west. After a first-placement drop or accepted keyboard target has staged a territory or wasteland target, the same arrow keys and `O` hotkey keep updating the pending orientation until confirmation. When browser init includes `?gnostica-dev-demo-hotkeys=1`, the command dialog also lists `D`, which rewrites only the dev app-db view so every configured deck card appears as a board territory.

Initial-placement target visuals stay focused: legal empty territories and wastelands remain valid drop targets, but the board paints the center-starting keyboard target or the currently hovered drag target instead of blanket-highlighting every legal square.

The `I` shortcut should preserve the current 3D OrbitControls camera view; only the `Reset view` control should reset the board camera.

## Issue Tracking

This repo tracks work with `br` from beads_rust. Start from `br ready` and `br show <id>` when picking up tickets; see `AGENTS.md` for the detailed workflow and command reference.

This repository is intentionally `br`-only. Do not use the legacy Go `bd` CLI here: the active tracker state is the br SQLite/JSONL pair under `.beads/`. A tracked regular file at `.beads/embeddeddolt` intentionally blocks legacy `bd` from auto-discovering an embedded-Dolt workspace and overwriting `.beads/issues.jsonl`; if a checkout still has a directory there, move or delete that stale cache before running tracker commands.

If `.beads/issues.jsonl` is accidentally removed by legacy tooling, recover it from the br SQLite database with `br sync --flush-only --force`, then verify with `br doctor` and `br sync --status`.

After updating issues, run `br sync --flush-only` and include the resulting `.beads/` changes in the current Jujutsu change or commit. `br` does not run git or Jujutsu commands for you.
